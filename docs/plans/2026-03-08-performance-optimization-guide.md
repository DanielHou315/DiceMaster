# Performance Optimization Guide

## Baseline Measurements (dice1, 2026-03-08)

System: Raspberry Pi CM4, 4 cores, 7.6 GiB RAM. Total CPU at runtime: ~240-270% of 400%.

| Node | %CPU | Root Cause |
|------|------|------------|
| chassis.py | 71-83% | 30 TF lookups/tick, Python quaternion math |
| game_manager.py | 51-59% | Unknown — unexpectedly high at idle |
| motion_detector.py | 40-41% | Publishes 50Hz regardless of state change |
| imu_hardware.py | 18-27% | 14 separate I2C reads per sample |
| screen_bus_manager (×3) | 12-15% each | Thread wake patterns, Python interpreter ×3 |
| madgwick_filter (C++) | 6-9% | Fine |
| robot_state_publisher | <1% | Fine |

Topic rates: `/imu/data_raw` 50Hz, `/imu/data` 45-50Hz, `/imu/motion` 50Hz, `/tf` 67Hz, `/chassis/orientation` 10Hz.

## Priority 1: Replace TF-Based Orientation with C++ Direct Computation

**Impact: chassis.py 71-83% → estimated <5%**

### Problem

`chassis.py` performs 30 `tf_buffer.lookup_transform()` calls per orientation tick (10Hz = 300 lookups/sec), each with a 10ms blocking timeout. All URDF joints are `type="fixed"` — every screen frame and edge frame has a constant transform relative to `base_link`. The entire TF tree traversal is wasted work.

### Solution: `dice_orientation_node` (C++)

A C++ ROS2 node that:
1. Subscribes to `/imu/data` (filtered quaternion from Madgwick)
2. Reads a YAML config file containing face normals, edge offsets, and the base joint transform (extracted from the URDF by a one-time Python script)
3. Computes all screen orientations via direct quaternion-vector multiplication
4. Publishes `ScreenPose` messages and `ChassisOrientation`
5. Optionally publishes the `world → imu_link` TF transform (replacing that function from the old chassis node)

#### Config file format (`dice_geometry.yaml`)

Generated by the `urdf_to_dice_config.py` script (see below). The config uses a **simplified representation** that exploits cube symmetry: only one canonical screen's edge geometry is stored. The 6 face orientations are defined by their joint quaternions, and the canonical edges are rotated per-face at load time.

```yaml
# Generated from dice.urdf.xacro — do not edit manually.
# Regenerate with: python3 urdf_to_dice_config.py <urdf_path> -o dice_geometry.yaml

base_joint:
  # Transform from imu_link to base_link (rpy in the URDF)
  quaternion: [qx, qy, qz, qw]

# Canonical screen edge offsets (from screen_component.urdf.xacro).
# These are in the screen-local frame and are identical for all 6 screens.
# The C++ node rotates these by each screen's joint_quaternion at startup.
canonical_edges:
  top:    [0.0475, 0.0, 0.001]
  right:  [0.0, -0.0475, 0.001]
  bottom: [-0.0475, 0.0, 0.001]
  left:   [0.0, 0.0475, 0.001]

# Half the cube side length. Face center offsets are derived from
# joint_quaternion applied to [0, 0, face_offset].
face_offset: 0.0508

screens:
  1:
    joint_quaternion: [qx, qy, qz, qw]  # from URDF base_link → screen_1_link
  2:
    joint_quaternion: [qx, qy, qz, qw]
  3:
    joint_quaternion: [qx, qy, qz, qw]
  4:
    joint_quaternion: [qx, qy, qz, qw]
  5:
    joint_quaternion: [qx, qy, qz, qw]
  6:
    joint_quaternion: [qx, qy, qz, qw]

parameters:
  rotation_threshold: 0.7
  edge_detection_frames: 2
  orientation_rate: 10.0
  publish_tf: true
```

At startup, the C++ node derives per-screen face normals and edge offsets in `base_link` frame from the joint quaternions and canonical geometry. These are stored as pre-computed arrays for SIMD processing.

#### SIMD-Optimized Computation

The Pi CM4 (Cortex-A72) supports ARM NEON, which processes 4×float32 in parallel. The orientation computation is structured to exploit this.

##### Startup: precompute geometry arrays (once)

```cpp
// For each screen, precompute face normal and edge offsets in base_link frame
// from joint_quaternion and canonical geometry.
//
// face_normals[6]  — unit vectors in base_link frame (derived from joint_q * [0,0,1])
// face_offsets[6]  — face center positions in base_link frame (joint_q * [0,0,face_offset])
// edge_offsets[6][4] — 4 edge positions per screen in base_link frame
//                      (joint_q * canonical_edge for each edge)
//
// Store as SoA (struct-of-arrays) for SIMD:
//   face_normals_x[8], face_normals_y[8], face_normals_z[8]  (padded to 8 for 2×NEON loads)
//   Similar for edge data.
```

##### Per-tick: SIMD batch orientation (10Hz)

**Phase 1: All 6 face z-positions in 2 NEON batches**

A quaternion rotation `q * v * q⁻¹` expands to a formula involving only the quaternion components and the vector. For a unit quaternion `q = (w, x, y, z)` rotating vector `v = (vx, vy, vz)`:

```
result.z = vz*(1 - 2*(x² + y²)) + vx*(2*(x*z + w*y)) + vy*(2*(y*z - w*x))
```

Since we only need the z-component to determine which face is up, we compute only `result.z` — 1/3 of the full rotation work.

```cpp
// q_world = q_imu * q_base  (one quaternion multiply, ~16 FLOPs)

// Batch-rotate all 6 face normals, extracting only z-component.
// NEON processes 4 faces at once:
//
// float32x4_t nx = vld1q_f32(face_normals_x);  // load normals[0..3].x
// float32x4_t ny = vld1q_f32(face_normals_y);
// float32x4_t nz = vld1q_f32(face_normals_z);
// float32x4_t z_out = compute_rotated_z(q_world, nx, ny, nz);  // 4 faces at once
//
// Repeat for faces [4..5] (padded to 4, ignore padding).
// Total: 2 NEON batches for 6 faces.
//
// Result: up_alignment[6] = z_out[i] / face_offset  (clamped to [-1, 1])
```

This replaces 6 sequential TF lookups with 2 NEON vector operations (~30 FLOPs total vs 6 TF tree traversals with mutex locks and interpolation).

**Phase 2: Top face edge positions — 1 NEON batch**

Only the top face (highest `up_alignment`) needs edge computation for rotation detection. The 4 edges of one screen fit exactly in one NEON batch:

```cpp
// Identify top screen from Phase 1 (argmax of up_alignment[6]).
// Only compute edges for that one screen.
//
// int top = argmax(up_alignment, 6);
// float32x4_t ex = vld1q_f32(edge_offsets_x[top]);  // 4 edges at once
// float32x4_t ey = vld1q_f32(edge_offsets_y[top]);
// float32x4_t ez = vld1q_f32(edge_offsets_z[top]);
// float32x4_t edge_z = compute_rotated_z(q_world, ex, ey, ez);
//
// Lowest edge_z → determines screen rotation (0°/90°/180°/270°).
```

This replaces 24 TF lookups (4 edges × 6 screens) with 4 (1 batch of 4 edges for the top screen only).

**Total per tick: 1 quaternion multiply + 3 NEON batches ≈ ~60 FLOPs.**

Compare to the current Python chassis: 30 TF lookups × (mutex acquire + tree walk + interpolation + timeout handling) ≈ thousands of operations plus blocking I/O.

##### Fallback path

If NEON is unavailable (e.g. running on x86 for development/emulation), the same logic runs with scalar Eigen math. The SIMD path is selected at compile time via `#ifdef __ARM_NEON`.

#### `urdf_to_dice_config.py`

A Python script that parses the URDF (via `urdf_parser_py` or raw XML) and extracts:
- `base_joint` RPY → quaternion
- Per-screen: joint `rpy` → `joint_quaternion`
- `face_offset` from joint origin `xyz` magnitude
- Canonical edge offsets from `screen_component.urdf.xacro`

The script validates cube symmetry assumptions (all faces equidistant, all edge sets identical) and warns if the URDF deviates.

Run once when the URDF changes (e.g. IMU location shifts in a new PCB revision):

```bash
python3 urdf_to_dice_config.py resource/dice.urdf -o config/dice_geometry.yaml
```

The C++ node reads `dice_geometry.yaml` at startup. No URDF parsing at runtime.

#### Relationship to `robot_state_publisher`

`robot_state_publisher` is **kept** in the launch file but only started in debug/development mode. It remains useful for:
- RViz visualization during development
- Debugging orientation issues with `tf2_echo`
- Validating that the C++ node's output matches the TF tree

In production, `robot_state_publisher` is not launched — the C++ node computes everything directly.

### Implementation plan

1. Write `urdf_to_dice_config.py` — parse URDF, emit simplified `dice_geometry.yaml` with canonical edges + per-screen joint quaternions. Validate symmetry assumptions.
2. Write `dice_orientation_node.cpp`:
   - Startup: load `dice_geometry.yaml`, precompute face normals and edge offsets in SoA layout for NEON
   - Runtime: subscribe to `/imu/data`, compute orientations via SIMD batch rotation
   - Phase 1: batch all 6 face normals → z-only rotation in 2 NEON passes
   - Phase 2: batch 4 edges of top face only → 1 NEON pass
   - Publish `ScreenPose` + `ChassisOrientation` + optional TF
   - Scalar fallback via `#ifdef __ARM_NEON` for x86 dev/emulation builds
3. Write `CMakeLists.txt` additions — compile with `-mfpu=neon` on ARM, link Eigen for fallback
4. Create `dice_orientation_node.launch.py` — replaces `chassis.launch.py` in production
5. Validate: run both old chassis node and new C++ node side by side, compare outputs on dice1
6. Update `dicemaster.launch.py` to use the C++ node by default, with a `debug` launch arg to switch back to Python + robot_state_publisher + full TF tree

## Priority 2: Investigate Game Manager Idle CPU (51-59%)

**Impact: 51-59% → expected <2%**

The game manager should be nearly dormant at idle. Suspected causes:
- A tight timer or spin loop inside the game strategy
- The `MultiThreadedExecutor` thread pool churning
- The default game's strategy doing continuous work

### Investigation steps

1. Profile with `py-spy` on dice1: `sudo py-spy top --pid $(pgrep -f game_manager)`
2. Check if the default game (`chinese_quizlet`) strategy has a timer or loop
3. Check `ShakeQuizletStrategy` for polling patterns in its motion subscription

### Likely fix

The strategy probably processes every `/imu/motion` message (50Hz) even when idle. Combine with Priority 4 (motion detector change-only publishing) to eliminate this.

## Priority 3: IMU Burst Read

**Impact: imu_hardware.py 18-27% → estimated 5-10%**

### Problem

`imu_hardware.py:263-308` — `_read_raw_data` calls `read_byte_data` twice per axis (high byte, low byte), 6 axes + temperature = 14 I2C transactions per sample at 50Hz = **700 I2C syscalls/sec**.

### Fix

Replace with a single burst read:

```python
# Before: 14 separate I2C reads
accel_x = self._read_raw_data(self.ACCEL_XOUT_H)  # 2 reads
accel_y = self._read_raw_data(self.ACCEL_YOUT_H)  # 2 reads
# ... 10 more reads

# After: 1 I2C read
data = self.bus.read_i2c_block_data(self.i2c_address, self.ACCEL_XOUT_H, 14)
accel_x = struct.unpack('>h', bytes(data[0:2]))[0]
accel_y = struct.unpack('>h', bytes(data[2:4]))[0]
accel_z = struct.unpack('>h', bytes(data[4:6]))[0]
temp    = struct.unpack('>h', bytes(data[6:8]))[0]
gyro_x  = struct.unpack('>h', bytes(data[8:10]))[0]
gyro_y  = struct.unpack('>h', bytes(data[10:12]))[0]
gyro_z  = struct.unpack('>h', bytes(data[12:14]))[0]
```

50 I2C transactions/sec instead of 700.

## Priority 4: Motion Detector — Change-Only Publishing

**Impact: motion_detector.py 40% → estimated 5-10%. Also reduces downstream load on game strategies.**

### Problem

`motion_detector.py:140-188` publishes a full `MotionDetection` message on **every** IMU callback (50Hz) regardless of whether the motion state changed. Also computes `shake_intensity` 3 times redundantly per tick.

### Fix

1. Compute `shake_intensity` once, pass to `detect_shaking` and `get_stillness_factor`
2. Publish only when state changes: `shaking` transitions (started/stopped), or at a reduced heartbeat rate (e.g. 2Hz) for liveness
3. Remove `print("Shake vals:", ...)` at line 94

## Priority 5: Disable Redundant Madgwick TF Broadcast

**Impact: eliminates 50Hz TF spam, reduces `/tf` topic from 67Hz to ~10Hz**

### Problem

`imu.launch.py:69` sets `publish_tf: True` on the Madgwick filter. The chassis node (or the new C++ orientation node) also broadcasts `world → imu_link`. Two sources publishing the same transform at different rates.

### Fix

Set `publish_tf: False` in `imu.launch.py`. The orientation node owns TF broadcasting.

## Priority 6: Fix GIF Rotation Bug

**Impact: functional correctness, not just performance**

### Problem

`screen.py:307` — `current_frame_messages[0].rotation = self.gif_rotation` only sets the Python attribute. The `payload` bytearray was pre-encoded at construction time and still contains the old rotation byte. The ESP32 receives stale rotation data. Same bug in `_resend_with_rotation` at line 332.

### Fix options

- Re-encode the message payload after mutation (re-serializes ~8KB per chunk)
- Better: send rotation as a separate lightweight SPI command, decouple rotation from image payload
- Best: encode lazily at transmission time instead of at construction time

## Priority 9: Use SingleThreadedExecutor for Simple Nodes

**Impact: eliminates thread pool overhead for 4+ nodes**

### Problem

Every `main()` uses `MultiThreadedExecutor`, which spawns a thread pool even for nodes with a single subscriber and no concurrent callback needs. The `pose_lock` in `chassis.py` exists only because `MultiThreadedExecutor` creates the race condition.

### Affected nodes

- `imu_hardware.py` — single timer, single service → `SingleThreadedExecutor`
- `motion_detector.py` — single subscriber, single publisher → `rclpy.spin(node)`
- `chassis.py` / new C++ node — single subscriber, two timers → `SingleThreadedExecutor` (or `rclpy.spin`)
- `energy.py` — two timers → `SingleThreadedExecutor`

Only `game_manager.py` and `screen_bus_manager.py` have legitimate multi-threading needs.

## Priority 10: GIF File I/O Optimization

**Impact: reduces game load time, eliminates redundant disk I/O**

### Problem

`media_types.py:497-532` — `GIF.to_msg()` ignores already-loaded `self.frames_data` and re-reads all frame files from disk via `os.listdir` + `Image(file_path=...)`. Also writes a temp file for PIL validation (`media_types.py:438-463`) when `io.BytesIO` would work.

### Fix

1. Use `self.frames_data` (already in memory) in `to_msg()`
2. Replace tempfile-based PIL validation with `PILImage.open(io.BytesIO(data))`
3. Fix hardcoded `8192` at line 479 to use `SPI_CHUNK_SIZE` constant
